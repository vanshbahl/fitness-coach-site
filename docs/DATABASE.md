# Database Design

## Overview
The database uses PostgreSQL for production and SQLite for local development. The schema is built to capture the rich data generated by the interactive onboarding assessment, moving beyond simple contact forms.

## Tables

### 1. `users` (Admin)
Stores information about the coach/admin.
- `id` (UUID, Primary Key)
- `email` (String, Unique, Indexed)
- `hashed_password` (String)
- `role` (Enum: `admin`)
- `created_at`, `updated_at` (Timestamps)

### 2. `trial_bookings` (The Assessment Model)
Captures the complete onboarding wizard payload. This table is wide but flattened for the MVP to allow easy querying and dashboard display.

#### Personal Information
- `id` (UUID, Primary Key)
- `name` (String) - Free text
- `age` (Integer) - Single value
- `gender` (Enum/String) - Single select
- `city` (String) - Free text
- `whatsapp_number` (String) - Free text
- `instagram_handle` (String, Nullable) - Free text

#### Body Metrics & Assessment
- `height_cm` (Integer) - Captured via visual slider
- `weight_kg` (Float) - Captured via visual slider
- `goals` (ARRAY of Strings / JSONB) - Multi-select (e.g., ["Build Strength", "Get Lean"])
- `fitness_level` (Enum: `beginner`, `intermediate`, `advanced`) - Single select interactive card
- `equipment_available` (ARRAY of Strings / JSONB) - Multi-select visual cards
- `preferred_training_days` (ARRAY of Strings / JSONB) - Multi-select chips
- `current_routine` (Text, Nullable) - Free text input

#### Booking & Payment Meta
- `scheduled_at` (Timestamp, Indexed) - Selected slot
- `status` (Enum: `pending`, `paid`, `completed`, `cancelled`, `enrolled`) - default `pending`
- `razorpay_order_id` (String, Nullable, Indexed)
- `razorpay_payment_id` (String, Nullable)
- `amount` (Integer) - Stored in paise (e.g., 4900)
- `calendar_event_id` (String, Nullable)
- `attendance` (Boolean) - default `false`
- `admin_notes` (Text, Nullable)
- `created_at`, `updated_at` (Timestamps)

*Note on Arrays*: Since SQLite does not natively support Postgres `ARRAY` types, we will use SQLAlchemy's `JSON` type for `goals`, `equipment_available`, and `preferred_training_days` to maintain compatibility between dev and prod, or serialize them as comma-separated strings if preferred.

## Relationships
- For the MVP, `trial_bookings` operates independently. Once the client portal is built, a `client_id` (Foreign Key) will link these assessments to registered user accounts.
