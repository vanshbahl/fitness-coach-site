# Database Design

## Overview
The database uses PostgreSQL for production and SQLite for local development. The schema is built to capture the rich data generated by the interactive onboarding assessment, moving beyond simple contact forms.

For the exhaustive list of all fields, data types, validation rules, and enums, refer to the **[Booking Domain Specification](BOOKING_DOMAIN.md)**.

## Tables

### 1. `users` (Admin)
Stores information about the coach/admin.
- `id` (UUID, Primary Key)
- `email` (String, Unique, Indexed)
- `hashed_password` (String)
- `role` (Enum: `admin`)
- `created_at`, `updated_at` (Timestamps)

### 2. `trial_bookings` (The Assessment Model)
Captures the complete onboarding wizard payload. This table is wide but flattened for the MVP to allow easy querying and dashboard display.

*Refer to [BOOKING_DOMAIN.md](BOOKING_DOMAIN.md) for the exact schema mapping of this table.*

#### Architectural Decisions
- **Multi-select Arrays vs JSON**: Since SQLite does not natively support Postgres `ARRAY` types, we utilize SQLAlchemy's `JSON` type for fields like `goals`, `equipment_available`, and `preferred_days`. This ensures 100% compatibility between the local SQLite environment and the production PostgreSQL environment.
- **Enums**: Strict SQLAlchemy Enums are used to prevent dirty data and enable precise analytics (e.g., `trial_outcome`, `fitness_level`).

## Relationships
- For the MVP, `trial_bookings` operates independently. 
- **Future Expansion**: Once the client portal is built, a `client_id` (Foreign Key to `users`) will link these robust assessments to registered user accounts, serving as the client's permanent fitness profile.
